## Successes and Strengths

1. **Splitting Story 5.5 out of Story 5.3 was the right call.** Story 5.3 was correctly scoped "visual-only" with an explicit "do NOT change how tool calls are stored or rendered" task constraint. Story 5.5 removed that constraint deliberately in a separate story — the data-model refactor (segmented union + JSONB column + SSE handler rewrites + backend dual-write) was always going to be ~5x the surface area of the visual-drift stories. Splitting prevented 5.3 from ballooning from 11 patches to a structural relocation. **Lesson worth carrying to future stories:** architectural rearrangements hiding inside visual ACs should be split at story-creation time, not discovered mid-implementation.

2. **Segments-based data model is a significant architectural improvement.** `MessageSegment` as a discriminated union (`{ type: 'text' } | { type: 'tool_call' }` with TS narrowing in handlers) is compiler-enforced type safety for inline-position state. Resume fidelity (NFR-R2) is improved — `Turn.segments` JSONB persists interleaved positional data alongside `content`, and the resume path reconstructs via `Array.isArray(turn.segments) ? (turn.segments as MessageSegment[]) : undefined`. This is the foundation Epic 6's `agent.service.ts` migration will build on.

3. **Bug-hunt found 0 critical, 0 high severity findings on the architectural refactor.** Story 5.5 rewrote every SSE event handler in `ConversationPane.tsx`, changed the `ChatMessage` data model, added a Prisma migration, and rewrote the backend `AgentService` + `AgentServiceFake` — and the bug-hunt found no production-reachable data-loss, security, or correctness regression introduced by the story. This is the strongest evidence yet that the ATDD red-phase scaffold practice + mockup-to-code mapping tables + DP-5 scope discipline work across visual-only AND architectural stories.

4. **Wave-1+2 closeout is the standing practice working as designed — twice.** Both runs caught what per-story review could not:

## Challenges and Growth Areas

### Recurring Patterns (from deep story analysis across all 5 stories)

1. **False-green tests: asserting presence without asserting the specific contract.** This is the recurring test-fidelity pattern across Epic 5. The 2026-07-12 retro flagged it once (5.3 AC-5 branded placeholder — test asserted only the default placeholder, not the wired-up branded variant). The 2026-07-13 bug-hunt surfaced it three more times:
   - **M1-new:** Story 5.5 AC-1 inline-position tests assert `textContent.toContain(...)` for each segment of `[Before tool, Bash, After tool]` — `textContent` concatenates ALL text nodes in DOM tree order, so any reordering of underlying segments would still leave all three substrings present and the test green. Same gap appears in AC-4, AC-5, AC-6, AC-9, and `AgentMessage.test.tsx` ordering test.
   - **L1-new:** `agent.service.spec.ts:184-202` asserts only `toHaveProperty('segments')` + `Array.isArray(...)` — a regression where the fake persists `segments: []` would still pass.
   - **L2-new:** `agent.service.unit.spec.ts:1413` asserts `toHaveProperty('status')` — verifies the field exists but not its value.
   - **Lesson consolidating across both runs:** every AC that specifies ordering, type values, or scoping must assert the *specific contract* (regex DOTALL for ordering, value comparison for fields, `arrayContaining`+`objectContaining` for shape), never just presence. A green test that asserts presence only is worse than no test — it provides false confidence.

2. **Architectural relocation inside a "visual-only" story is not a "visual-only change."** Carried forward from the 2026-07-12 retro (Story 5.3 AC-3 spinner/retry relocation). The fix landed as the Wave-1 M1 quick-win (extending the `useEffect` deps array at `ChatMessageList.tsx:45` to `[messages, isThinking, errorMessage, showRetry, showSpinner]`). Story 5.5 then went further — every SSE handler in `ConversationPane.tsx` was rewritten, with the new segment-based API introducing new state mutation paths. The bug-hunt caught one real-but-currently-unreachable status-overwrite (M2-new) downstream of an out-of-order `TOOL_CALL_RESULT`/`TOOL_CALL_END` sequence — the kind of edge case that only an adversarial reviewer walking every branching path would surface.

3. **Test-seam fakes must mimic production's side-effect timing.** Story 5.5 introduced M3-new (`AgentServiceFake` diverges from production's `pendingClassifierPromises` pattern). Production fires working-tree checks as fire-and-forget promises queued to a `pendingClassifierPromises` array, then `await Promise.allSettled(promises)` at end-of-turn before `RUN_FINISHED`. The fake awaits inline — blocking the event loop before the next scripted event. Result: any production bug depending on the timing relationship between `WORKING_TREE_DIRTY` and subsequent `TEXT_MESSAGE_*` events would pass fake-based tests but fail in production. Per `project-context.md:138`, "test-seam fakes mimic production side effects" — this is a documented convention, not a new rule. **Lesson:** test-seam parity is a story-creation-time testability concern, not a post-hoc bug-hunt finding; the fake's `runTurn` loop should have been reviewed for the `pendingClassifierPromises` pattern when Story 5.5 added the working-tree path.

4. **TypeScript narrowing bugs pass CI silently when the project has no typecheck nx target.** The M2-new fix's TS narrowing bug (see Success #7) passed `yarn nx test web` (jest transpiles via babel, no strict type-check) and `yarn nx lint web` (eslint doesn't catch TS narrowing). Only direct `npx tsc --noEmit -p apps/web/tsconfig.json` surfaced it. `apps/agent-be` has a `typecheck` nx target — `apps/web` does not. The bug shipped-and-was-caught only because the NFR audit happened to run `tsc` directly. Production impact: TypeScript narrowing errors are caught only at next.js production build time, not during development or pre-merge CI. **Lesson:** every project exposing a `typecheck` nx target should also expose it in `web`; the absence is a structural gap, not a one-off.

5. **Adjacent-code drift remains invisible until adversarial review.** The 2026-07-12 retro flagged this for `conversations/[conversationId]/loading.tsx` (canonical header pattern missed during 5.2's review). It is STILL OPEN. Story 5.5's scope touched neither file. The fix is a simple header restructure but requires verifying whether `<Breadcrumb />` is SSR-compatible inside `loading.tsx`. **Lesson:** pattern establishment needs an explicit all-files map at story-creation time and a completion audit, not case-by-case review — established as a team agreement in the prior retro; this retro reaffirms it.

6. **Code-level duplication in handler pairs invites future divergence.** Story 5.5's `MANUAL_SAVE_SUCCEEDED` and `MANUAL_SAVE_FAILED` handlers (~150 lines combined) share ~90% of their structure: parse toolCallId, optional fallback UUID, clear timeout ref, find target message, insert or append segment, set working-tree state. The only differences are `status` (`'completed' | 'error'`), `semantic` field (succeeded only), `errorMessage` field (failed only). Today the duplication is symmetric and well-tested. A future edit to one handler without the other risks behavioral divergence that's hard to spot in review. (L6-new Low; bug-hunt Recommendation: extract a `buildManualSaveSegment` + `insertManualSaveSegment` helper.)

## Key Insights and Learnings

1. **A visual-drift epic can harbor an architectural story without becoming incoherent — but only when the architectural story is split explicitly.** Story 5.5's scope (data-model refactor + SSE handler rewrite + Prisma migration + backend dual-write) is qualitatively different from Stories 5-1..5-4 (CSS-class alignment + token guardrails). Splitting at story-creation time preserved the visual-drift epic's velocity (Stories 5-1..5-4 took a single day collectively) AND let the architectural story ship clean with its own bug-hunt + NFR pass.

2. **Running the Wave-1+2 closeout twice is not redundant; it is the standing practice at the right granularity.** The first closeout surfaced 11 findings for the 4 visual stories; the second surfaced 9 new findings for the architectural story. The instrument scales: bug-hunt + traceability + NFR + deferred-work prune per closeout wave. **Lesson:** invoke Wave-1+2 at every "natural closeout boundary" — end of epic, end of architectural story, end of milestone. The team agreement from the 2026-07-12 retro is now confirmed by repetition.

3. **The segments-based data model unblocks Epic 6 with low migration cost.** Story 5.5's dual-write (`accumulatedText` for backward compat + `segments` for inline rendering) preserves backward compatibility with both the `content` field and the legacy single-Markdown render path. Epic 6's `agent.service.ts` migration can switch the primary write path to `segments` without coordinating a breaking schema change. This is the architectural payoff of doing 5.5 *before* Epic 6 rather than folding it into 6's first story.

4. **Test-fidelity false-greens are a recurring Epic 5 pattern, not a one-off.** Three distinct shapes appeared:
   - **Ordering scoping (5.3 AC-5 / 5.5 AC-1, AC-4, AC-5, AC-9, AgentMessage):** tests asserting textContent presence instead of relative position.
   - **Shape-only assertion (5.5 L1-new):** test asserts `Array.isArray(segments)` instead of segment contents.
   - **Field-existence without value (5.5 L2-new):** `toHaveProperty('status')` instead of `.toBe('completed')`.
   The NFR audit fixed all three via quick-wins during run #2: regex DOTALL for ordering, `arrayContaining`+`objectContaining` for shape, `.toBe('completed')` for value. **Lesson worth codifying:** a green test asserts the specific contract, not the presence of the contract's bytes.

5. **Test-seam fakes that diverge from production's side-effect timing are a class of test-fidelity debt that compounds silently.** Story 5.5's `AgentServiceFake` diverges from production's `pendingClassifierPromises` pattern (M3-new). Today it is more deterministic than production; tomorrow it could mask a timing-dependent UI bug that ships to production uncaught. The fake's `runTurn` loop should be reviewed at story-creation time against production's promise-queue patterns, not at bug-hunt time.

6. **CI gates should include direct `tsc --noEmit` when the test runner babel-transpiles.** `yarn nx test web` (jest via babel) and `yarn nx lint web` (eslint) both reported 0 errors on the M2-new fix — the TS narrowing bug only surfaced via `npx tsc`. `apps/agent-be` has `typecheck` as an nx target; `apps/web` does not. Adding the target is a 5-line `project.json` change with high amortized value.

7. **Bug-hunt quick-wins are an effective follow-through mechanism for prior-wave open findings.** The 2026-07-13 bug-hunt (run #2) closed 2 prior-wave mediums (M1 auto-scroll, M2 no-scrollbar pane) and 3 prior-wave lows (L2 aria-live assertion, L4 Intl hoist — verified already-fixed-before; L7 maxLength — production fix had landed without test, test added in this run). The bug-hunt as an instrument doesn't only find new issues — it verifies status of prior findings and applies small inline fixes when they're trivial. **Lesson:** carry forward the "verify + quick-win-apply" approach in every Wave-1+2 closeout run.

8. **Story 5.5 deviated from the prior retro's recommendation (fold into Epic 6 backlog) — and that was correct.** The 2026-07-12 retro recommended folding Story 5.5 into Epic 6's planning, reasoning that the architectural scope (data-model refactor + SSE handler rewrites + Persistence format changes + ConversationPane.tsx changes) overlapped with Epic 6's planned `agent.service.ts` migration. The team's decision to implement Story 5.5 as Epic 5's fifth story (rather than deferring to Epic 6) preserved the epic's atomicity: the conversation-stream UX surfaces were closed at mockup fidelity INCLUDING the architectural depth, and Epic 6 can pick up the segments-based model AS A GIVEN rather than refactoring it mid-migration. **Lesson:** when an architectural story is naturally a prerequisite for the next epic, doing it in the current epic (with its own Wave-1+2 closeout) is the right sequencing — Epic 6 starts with the new data model already in place.

## Key Insights from this Retro

1. **Splitting Story 5.5 out of Story 5.3 at story-creation time was the right call.** Architectural rearrangements hiding inside visual ACs should be split — not discovered mid-implementation. The data-model refactor (segmented union + JSONB column + SSE handler rewrites + backend dual-write) shipped cleanly because it had its own ATDD scaffold, its own bug-hunt, its own NFR audit.

3. **The segments-based data model is a significant architectural improvement and is the substrate Epic 6 will build on.** The discriminated union (`{ type: 'text' } | { type: 'tool_call' }` with TS narrowing in handlers), the JSONB `Turn.segments` column, and the dual-write (`accumulatedText` + `segments`) preserve backward compatibility while unblocking inline-pill rendering at the exact stream position of the tool call (UX-DR5 contract).

The 3 new Medium findings (M1-new false-green ordering tests, M2-new status-overwrite, M3-new test-seam divergence) are all test-fidelity concerns masking FUTURE regressions, not bugs today.

The codified team agreement: a green test asserts the specific contract, never just presence.
6. **The `AgentServiceFake` divergence (M3-new) is a test-fidelity concern that should be addressed before Epic 6.** The fake awaits the working-tree check inline; production fires it as a `pendingClassifierPromises` fire-and-forget. Any production bug depending on the timing relationship between `WORKING_TREE_DIRTY` and subsequent `TEXT_MESSAGE_*` events would pass fake-based tests but fail in production. Epic 6's 6.2/6.3 will exercise this timing under sandbox execution — close it first.
7. **The 2026-07-12 "fold Story 5.5 into Epic 6" recommendation was overridden correctly.** Implementing Story 5.5 in Epic 5 (rather than deferring to Epic 6) preserved Epic 5's atomicity AND let Epic 6 pick up the segments-based model AS A GIVEN. Doing an architectural story as a prerequisite *before* the next epic beats folding it into the next epic's migration.
8. **TypeScript narrowing bugs pass CI silently when the project has no `typecheck` nx target.** The M2-new fix's TS narrowing bug passed `yarn nx test web` (jest via babel) and `yarn nx lint web` (eslint) — only direct `npx tsc --noEmit` surfaced it. `apps/agent-be` has the target; `apps/web` doesn't. The structural gap is closed by a 5-line `project.json` change.
