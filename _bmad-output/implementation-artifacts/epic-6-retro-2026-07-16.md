## Challenges and Growth Areas

### Recurring Patterns (from deep story analysis across all 5 stories)

1. **Real-service verification was written but could not run — the single largest gap in Epic 6.** Stories 6.4 and 6.5 are correctly scoped as "verification, not implementation" stories. But the verification could not execute:
   - **Story 6.5 real-service specs** (5 new functional specs + existing NFR performance + manual commit specs) require real Daytona + real Claude Code + real Anthropic API + real GitHub OAuth. The operational prerequisites (GitHub test account with 2FA, `TEST_GITHUB_*` CI secrets, real env vars in `.env.local`) are human-action items, not dev tasks.
   - **PR-tier browser tests** (3 restored hover blocks from Story 5.4, 1 auto-scroll regression from Epic 5 P5) are marked `test.fixme()` due to a JWT Edge-vs-Node.js decryption issue — the middleware (Edge runtime) cannot decrypt the JWT encoded by `next-auth/jwt` `encode()` in Node.js. Browser pages redirect to `/sign-in`.
   - **`webServer` port conflict** — `reuseExistingServer: false` for the real-service tier conflicts with already-running dev servers.
   - **OAuth Configuration error** — `AUTH_GITHUB_ID`/`SECRET` mismatch or callback URL misconfigured; `realOAuthFlow()` button click times out.
   - **Outcome:** Epic 6 is marked `done` in sprint-status without the real-service E2E verification ever having run. The migration is structurally sound (789 agent-be tests + 908 web tests pass, architecture is followed) but operationally unverified. **Lesson:** when a story's entire value proposition is "verification," not being able to run the verification is a fundamental epic-level gap, not a deferred finding.

2. **The sandbox-agent invocation command was unknown across 3 stories.** Story 6.2 research (Task 1) discovered sandbox-agent is an HTTP server (Rust daemon on port 2468), not a stdout-emitting CLI. This contradicts the architecture's assumption (line 668: "sandbox-agent JSONL → event bridge → SSE → browser"). The event bridge was built to parse JSONL on stdout as the ATDD contract required — the exact invocation command that produces this output was deferred from 6.2 → 6.3 → 6.5. Story 6.3 constructed `sandbox-agent --agent claude-code --prompt <message>` as a reasonable placeholder and flagged it for Story 6.5 verification. Story 6.5 could not verify it. **Lesson:** SDK discovery should happen before story creation, not during implementation. The architectural assumption about the real binary's behavior was not validated against the binary before story creation.

3. **Cost data gap persists across the epic.** Story 6.2 flagged that sandbox-agent's `AgentEvent` schema (`{ event_id, session_id, type, data }`) does not document cost data as an AG-UI event type. Story 6.3 implemented cost capture from `RUN_FINISHED` event's `data` payload — if sandbox-agent surfaces cost data there, `CostTrackingService` persists it. If it doesn't, cost tracking is a silent no-op. Whether cost data is actually surfaced cannot be verified without a real sandbox-agent session (Story 6.5, which cannot run). NFR-O1 (per-user LLM spend tracking) may be silently broken for sandbox-based execution. **Lesson:** when a data-flow gap is flagged early and cannot be resolved without real-service verification, the gap should be tracked as an epic-level risk item, not just a story-level deferral.

4. **`networkAllowList` egress-blocking activation is unverified.** Story 6.1 used `0.0.0.0/32` (a dummy CIDR) to force egress-restriction activation. Daytona's docs say all tiers pre-whitelist package registries, GitHub/GitLab, container registries, and AI/ML APIs — but whether the pre-whitelist is correct, whether `0.0.0.0/32` actually activates restriction, and whether the agent's `npm install` (which `installBinaries()` depends on) is reachable through the pre-whitelist — all unverified. Story 6.5's negative egress test (`curl https://example.com` should fail) is written but cannot run. **Lesson:** security-critical behavior that depends on an external service's (Daytona's) undocumented internal behavior should not be assumed.

5. **Pre-existing F4 sibling bug in `injectGitConfig()` was found by NFR audit but deferred.** Story 6.4 fixed the empty-error-message bug in `SandboxService.commit()` (F4: `throw new Error(response.result)` → `throw new Error(response.result || ...)`. The NFR audit found the same bug exists in `injectGitConfig()` (`sandbox.service.ts:192-193, 201-202`) — 30 lines above the F4 fix. The existing `injectGitConfig()` tests mock non-empty `result` strings, giving false confidence — they never exercise the empty-`result` path that occurs in reality (`git config` in a non-git directory exits 128 with empty stdout and `fatal:` on stderr). This was deferred (pre-existing, not story-introduced) but it's the same SDK boundary, same file, same class of bug. **Lesson:** when a fidelity audit finding reveals a class of bug, the sibling instances in the same file should be fixed in the same story, not deferred.

6. **Blind Hunter review layer was skipped (config change `3eaba57`).** Every Epic 6 story's code review ran Edge Case Hunter + Acceptance Auditor, with Blind Hunter skipped per the config change `3eaba57` (`skip blind hunter review layer`). For an epic of this architectural significance — removing the primary agent execution SDK and replacing it with a completely different transport — the skip of Blind Hunter is a gap. Blind Hunter's value is perspective-blind adversarial review (it doesn't see the story spec, only the code changes), which catches issues that Acceptance Auditor (spec-driven) and Edge Case Hunter (path-driven) don't. **Lesson:** for architecture-correction epics, the Blind Hunter layer should be re-enabled — the cost of a missed issue in a primary-transport rewrite is higher than in a feature story.

## Key Insights and Learnings

1. **A known deviation, documented at story time and corrected in a dedicated epic, is the right way to handle architecture-prescribing constraints that can't be met immediately.** Story 3.3's DP-2 deviation was visible, documented, and had a clear correction path (Epic 6). The 3-story delay between the deviation (3.3) and the correction (6.1–6.3) was acceptable because the deviation was explicit, not silent. **Carry forward:** when architectural constraints can't be met at story time, document the deviation explicitly and schedule the correction — don't silently ship a non-conforming implementation.

2. **The event-vocabulary preservation is the key to transport migration without side-effect refactoring.** Story 6.3 replaced SDK-specific message types (`assistant`, `user`, `result` — SDK's internal format) with AG-UI events (`TEXT_MESSAGE_CONTENT`, `TOOL_CALL_START`, `TOOL_CALL_RESULT` — transport-agnostic). Everything downstream — SSE pipeline, tool-pill classifier, cost tracking, pending classifier promises, working-tree emission — consumed AG-UI events and continued to work without changes. The SSE event contract was the stable boundary; the transport beneath it was replaceable. **Lesson:** when migrating a primary transport, identify the stable event vocabulary above the transport and preserve it — the migration becomes "swap the engine, keep the API."

3. **When verification stories can't verify, "done" is a planning-status, not a quality-status.** Stories 6.4 and 6.5 are correctly marked `done` — their code changes are complete, their tests pass at the unit/integration level, their ACs are structurally satisfied. But the real-service E2E verification — the actual confirmation that sandbox-based execution works against a real Daytona sandbox — has never run. This is a different kind of "done" from the architectural stories (6.1–6.3). **Lesson:** distinguish "structurally done" (code written, unit tests pass, ACs satisfied) from "operationally validated" (real-service E2E passes). Track the gap explicitly; don't let "done" in sprint-status mask an unverified verification.

4. **SDK discovery before story creation prevents mid-implementation surprises.** Story 6.2's Task 1 research discovered sandbox-agent is an HTTP server, not a stdout-emitting CLI — a fundamental discovery about the binary the architecture assumed would emit JSONL on stdout. This discovery was handled well (DP-2: architecture wins, contract is preserved, exact command deferred), but it should have been known before the story was created. The architecture's assumption (line 668) was not validated against the real binary during epic planning. **Lesson:** for epics that depend on external binaries or services, a pre-epic spike validating the architecture's assumptions against the real artifact prevents story-level discovery.

5. **Pre-epic fidelity audits are high-ROI.** The CF3 SandboxService fidelity audit identified 5 findings before any Epic 6 story was implemented. Distributing them across the right stories (F1–F3 in 6.1, F4–F5 in 6.4) meant no story had a mid-implementation "oh, we should also fix this" scope expansion. Compare to Epic 5, where Story 5.3's structural relocation was discovered mid-implementation. **Lesson:** pre-epic fidelity audits for files the epic will touch are as valuable as pre-epic architectural reviews — they surface test-confidence gaps before the dev is mid-flight.

6. **The ATDD red-phase scaffold workflow is now the standing practice and it works across architectural epics.** Every Epic 6 story had pre-created skipped tests generated by a separate prepare-tests workflow. The developer activated scaffolds (removed `.skip`) rather than creating tests. This worked for all 5 stories — including Story 6.3's complete rewrite of `AgentService.runTurn()` (19 SDK-based tests replaced with 45 sandbox-based tests) and Story 6.2's new service (44 tests across 2 spec files). The workflow produces well-structured tests with `[P0]`/`[P1]` priority tags and clear AC mapping. **Confirmed by repetition across Epic 5 + Epic 6** — the ATDD scaffold workflow is now a standing convention, not an experiment.

7. **The `onEvent` callback pattern (DP-3, Story 6.3) is a clean seam for "observe-and-react" in async event streams.** Three approaches were considered: (1) subscribe to `SessionEventsService`'s `ReplaySubject` (rejected — replay double-processing), (2) merge `AgentService` and `AguiEventBridgeService` (rejected — god-class), (3) add `onEvent` callback (chosen — simplest, no subscription, no replay issues, no circular dependencies). The callback is optional — the event bridge works without it (backward compat). Lifecycle event ownership prevents double-emission: when `onEvent` is provided, the bridge skips `sessionEvents.emit()` for lifecycle events (`RUN_STARTED`, `RUN_FINISHED`, `RUN_ERROR`); AgentService owns lifecycle emission. **Lesson:** for "observe-and-react" in async pipelines, a callback parameter is simpler than a subscription/replay pattern and avoids the double-processing trap.

## Significant Discoveries and Epic Update Recommendations

### Significant Change #1: sandbox-agent is an HTTP server, not a stdout-emitting CLI

**Evidence:** Story 6.2 Task 1 research found sandbox-agent (rivet-dev/sandbox-agent v0.4.2) runs as a Rust daemon (`sandbox-agent server`) exposing an HTTP + SSE API on port 2468. It does NOT emit a normalized event stream to stdout when invoked as a plain command. This contradicts the architecture's assumption (line 668: "sandbox-agent JSONL → event bridge → SSE → browser") that sandbox-agent emits JSONL to stdout.

**Decision (DP-2, Story 6.2):** The architecture mandates the pull-based `getSessionCommandLogs` transport. The event bridge was built to the ATDD contract (parse JSONL on stdout). The exact sandbox-agent invocation that produces JSONL-on-stdout was deferred from 6.2 → 6.3 → 6.5. Story 6.3 constructed `sandbox-agent --agent claude-code --prompt <message>` as a placeholder. Story 6.5 could not verify it.

**Impact on Epic 6:** structural — the event bridge is transport-agnostic (parses whatever JSONL it receives on stdout), but the actual sandbox-agent invocation that produces the expected output is unverified. If no such invocation exists (sandbox-agent is strictly an HTTP server), the event bridge's `onStdout` callback may never receive valid JSONL — the `sandbox-agent server` command would start the server, and `getSessionCommandLogs` would stream the server's startup logs, not agent events.

**Impact on Epic 7:** none directly (Epic 7 is frontend presentation only, independent of Epic 6).

**Epic update required:** YES — the architecture's data flow assumption (line 668) needs validation against the real sandbox-agent binary. If the JSONL-on-stdout contract is unattainable, the event bridge's transport mechanism (Daytona process session → `getSessionCommandLogs` → stdout callback) needs to switch to sandbox-agent's HTTP SSE API. This is a post-Epic-6 architectural concern — the event bridge's abstraction (parse JSONL on stdout) may need to be replaced with HTTP client SSE consumption. Recommend: (1) validate the sandbox-agent invocation against a real sandbox, (2) if the contract is unattainable, file a sprint change proposal for the transport switch.

## Key Insights from this Retro

1. **A known deviation, documented at story time and corrected in a dedicated epic, is better than forcing an incomplete architecture at story time.** Story 3.3's DP-2 was explicit and correctable. Epic 6 is the disciplined correction — 5 stories, tight sequencing, zero production-blocking regressions.

2. **The event-vocabulary preservation is the key to transport migration without side-effect refactoring.** Story 6.3 replaced SDK-specific message types with AG-UI events. Everything downstream (SSE pipeline, tool-pill classifier, cost tracking, working-tree emission) consumed AG-UI events and continued unchanged. The SSE event contract was the stable boundary.

3. **"Structurally done" is not "operationally validated."** Epic 6 is structurally complete — code, unit tests, ACs, code review, NFR audits all pass. But the real-service E2E verification has not run. The migration's correctness at the unit/integration tier is strong evidence, but the real sandbox-agent behavior, real Daytona transport, and real NFR latency are unverified. The gap must be tracked explicitly, not masked by "done."

4. **Pre-epic fidelity audits are high-ROI for files the epic will touch.** The CF3 audit identified 5 findings before any story was implemented. Distributing them across the right stories prevented mid-implementation scope expansion. This is now the standing practice for any epic touching tested code paths.

5. **SDK discovery should happen before story creation, not during implementation.** The sandbox-agent-is-an-HTTP-server discovery (Story 6.2 Task 1) should have been a pre-epic spike finding. The architecture's assumption about the binary's behavior was not validated during epic planning.
