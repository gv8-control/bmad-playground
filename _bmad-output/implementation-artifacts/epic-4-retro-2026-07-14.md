## Challenges and Growth Areas

### Recurring Patterns (from deep story analysis across all 12 stories)

1. **"Deferred as human action items" smell — recurred 5 times (4-5, 4-6, 4-7, 4-9, 4-10).** Story 4.5 deferred Tasks 4-6 (Vercel env vars, Railway env vars, OAuth App callback URL) as "infrastructure work requiring human action." These then:
   - Blocked Story 4.6's manual E2E verification (Tasks 4.6-4.8) — the agent-be container would crash on startup without `DATABASE_URL`, `ANTHROPIC_API_KEY`, `AUTH_SECRET`.
   - Forced Story 4.7's Task 1.1 to set `AUTH_SECRET`, `ANTHROPIC_API_KEY`, `CREDENTIAL_ENCRYPTION_KEK` on Railway via `railway variables set` — work that was 4.5's scope.
   - Were partially superseded by Story 4.9 (the OAuth App callback URL was never updated to `*.vercel.app`; Story 4.9 updates it directly to the custom domain per DP-3).
   - Story 4.10 noted "RAILWAY_TOKEN in `.env.local` is not authorized" — API verification pending human execution with a valid token.

   **Root cause:** "External service call with side effects" is correctly escalated per decision policy — but the deferral pattern let the work float between stories with no single owner, instead of being either (a) re-classified as a prerequisite the human completes before the story starts, or (b) explicitly assigned as a tracked action item with an owner and deadline.

2. **Test Pipeline failures blocked deploy verification — 3 stories (4-6, 4-7, partially 4-8).** Story 4.5's code shipped with lint errors in `CredentialErrorBanner.test.tsx` and typecheck errors in `anthropic-proxy.controller.spec.ts`. Story 4.6's deploy workflow correctly blocks deploys via its quality gate — exactly as designed. But the consequence was that Tasks 4.6-4.8 (Vercel/Railway deploy verification) were never completed in Story 4.6, and Story 4.7 had to fix the CI issues itself (PRs #27, #28) to unblock the deploy path. **Lesson:** a story marked `done` must include a green Test Pipeline run, not just "code committed." The Epic 2 retro action item #1 ("commit-before-done discipline") needs extension to "green-pipeline-before-done discipline."

3. **`project-context.md` rules adopted mid-epic but not back-applied — 3 runbooks (4.7, 4.8, 4.9).** The `--fail` and `--max-time 30` curl-flag requirement was first flagged in Story 4.9's NFR audit (deferred-work.md lines 473-474), then codified as a `project-context.md` rule (line 252). Stories 4.10 and 4.11 applied the rule from creation. But the earlier runbooks (`http2-verification.md` from 4.7, `deploy-failure-recovery.md` from 4.8, `custom-domain-setup.md` from 4.9) remain non-compliant — deferred findings in `deferred-work.md` that compound across three stories. Same pattern as Epic 2's "take: 100 silently truncates" — a deferral that recurs is avoidance.

4. **Railway CLI authentication was inconsistent across the epic.** Story 4.2 noted: "The Railway CLI's `RAILWAY_TOKEN` env var expects a project-level token, not an account-level token." Workaround: GraphQL API direct calls. Story 4.3 used `railway up` with a project-level token (provided by user). Story 4.8 noted: "RAILWAY_TOKEN env var is an API token (for GraphQL), not a CLI token (requires `railway login` OAuth flow)" — so `railway deployment redeploy` returned "Unauthorized." Story 4.10 noted: "RAILWAY_TOKEN in `.env.local` is not authorized — API returns 'Not Authorized'." The token management story is scattered across four stories without centralized documentation. **Lesson:** platform-specific auth quirks should be documented in `project-context.md` once discovered, not re-discovered story by story.

5. **False-green tests recurred — 3 stories (4-2, 4-6, 4-12).** The Epic 5 retro team agreement ("a green test asserts the specific contract, not just presence") was not fully applied:
   - 4-2: `expect(vars).toHaveProperty('DATABASE_URL')` prints all secrets on failure — flagged by NFR audit, deferred to 4.3, deferred again, ultimately migrated via `Object.keys()` in Story 4.5 (3 deferrals).
   - 4-6: `deploy-workflow.spec.ts` had vacuous P0 tests — `if` guards made `expect` unreachable (false-green). Patched in review.
   - 4-12: multiple false-greens in round 2 review — title format checked word presence not full pattern; section heading tests matched substring not heading structure; interval tests checked number presence not association with secret categories; due date value never asserted, only format. All patched.

6. **`generatePackageJson: true` produces insufficient runtime `package.json` — Story 4-3, DP-2 amendment.** The Dockerfile initially failed with `MODULE_NOT_FOUND` for `@prisma/client-runtime-utils` (Nx `generatePackageJson` doesn't include Prisma runtime deps from `@bmad-easy/database-schemas`), then `ws` (peer dep of `isomorphic-ws` from `@daytonaio/sdk`). Fix: merge ALL root `dependencies` into the generated `package.json` and copy the root `yarn.lock`. This is a known limitation of `generatePackageJson: true` for monorepos with transitive runtime deps — should be documented in `project-context.md` to prevent re-discovery.

7. **n8n workflow unrelated change leaked into Story 4.7's working tree.** The Story 4.7 review found a model swap (`voxtral-small-latest` → `mistral-small-latest` + version bumps) in `n8n/workflows/3D8Jw6GicWiwBQc6.json` that was not in any task, File List, or Debug Log Reference. Review deferred it as "Should be reverted from working tree before committing the story (DP-5: scope temptation)." Story 4.7's second review round found it was NOT reverted — patched. **Lesson:** review-time deferrals that say "revert before commit" must have a verification step at story-closing time.

8. **Required reviewers feature deferred past MVP — Story 4.6, project owner decision.** The GitHub billing plan for `gv8-control/bmad-playground` returned HTTP 422: "Failed to create the environment protection rule. Please ensure the billing plan supports the required reviewers protection rule." AC-3 is partially satisfied: branch restriction ✓, required reviewers ✗ (deferred past MVP). This is a real cost-of-doing-business finding — not a process defect, but worth recording as a known limitation.

## Key Insights and Learnings

1. **A "deferred as human action items" deferral is structurally different from a "deferred to a future story" deferral — and it's the worse of the two.** "Deferred to a future story" creates a tracked entry in `deferred-work.md` with a specific finding location. "Deferred as human action items" creates five separate floating tasks with no single owner, no clear deadline, and a strong tendency to be re-discovered piecemeal in the next story's prerequisite check. The Story 4.5 → 4.6 → 4.7 cascade is the canonical anti-pattern. **Lesson:** "human action items" deferrals must either (a) be re-classified as story prerequisites the human completes before the story starts, (b) be explicitly tracked as an action item with an owner and deadline, or (c) be escalated to Marius with a "blocked-on-human-action" flag. They must NEVER be silently deferred to "the next story will handle it."

2. **The Wave-1+2 closeout team agreement from the Epic 5 retro was NOT applied to Epic 4.** The Epic 5 retro reaffirmed: "Wave-1+2 closeout assessment runs at every natural closeout boundary — epic, architectural story, milestone." But Epic 4 has **no bug-hunt run, no epic-level traceability matrix update, no epic-level NFR assessment, no deferred-work prune at epic closeout.** The test-artifacts directory contains `atdd-checklist-4-*` files and per-story `nfr-assessment-4-*.md` files, but no `nfr-assessment-4-epic.md` or `bug-hunt-epic-4-*.md`. This is the most consequential process gap — the team agreement was not operationalized as a checklist item at epic-closeout time. **Lesson:** "team agreements" need to be backed by an automation hook or a script that emits a CHECK ITEM at epic-closeout time, not just recorded in a retro document.

3. **The "runbook + regression guard test" pattern is now a mature asset for infrastructure/deployment stories — and should be codified as a story-creation-time template.** Six consecutive stories (4-7 through 4-12) applied it with increasing sophistication. The pattern's value compounds: each new runbook story inherits the prior story's `CREDENTIAL_ENV_VARS` list, Bearer regex, curl-flag assertions, throw-on-missing-file `loadRunbook()` helper, `path.resolve(__dirname, '../../../../docs/runbooks/...')` locator, and `[P0]` tag conventions. **Lesson:** for any future story whose primary deliverable is a committed `docs/runbooks/*.md` file, the story spec should reference the canonical `monitoring-setup.spec.ts` or `secret-rotation-schedule.spec.ts` pattern explicitly at story-creation time, instead of each dev recreating the test scaffold from scratch.

4. **API-first provisioning is scriptable, idempotent, and self-documenting — but platform-specific auth quirks must be documented centrally, not re-discovered per story.** Story 4.1's Vercel REST API worked cleanly. Story 4.2's Railway GraphQL API needed workarounds (account token vs project token vs CLI token). Story 4.10's `RAILWAY_TOKEN` was "not authorized" for the volume backup API. These quirks are scattered across four story files without a single source of truth. **Lesson:** platform-quirk discoveries should be added to `project-context.md`'s "Infrastructure & Deployment" section as soon as the dev encounters them, not deferred to "future stories will document it."

5. **DP-2 amendments (spec vs code contradiction) recurred 4 times — each correctly resolved, but the pattern suggests spec authoring is not consulting the actual code at story-creation time.** 4-3: `generatePackageJson: true` insufficient. 4-4: AC said "three migrations," actual was nine. 4-12: deferred finding about `ANTHROPIC_API_KEY` not in env validation was STALE. 4-12: due-date formula used `ceil` when the AC intent required `floor`. **Lesson:** story-creation time should include a "verify the spec claims against current code" step — at minimum for any AC that references an exact count, a schema field, an API endpoint's request shape, or a runtime behavior.

6. **Story 4.12's two-round review is the new baseline for "runbook + cron workflow + config file + script" stories.** Round 1 caught 14 patches. Round 2 (a fresh-context re-review) caught 12 MORE patches — including a `null` entry crashing the secrets-array script, `gh issue create` failures aborting remaining secrets under `set -eo pipefail`, missing `concurrency:` block allowing duplicate issues, `gh issue list` failure silently falling back to `[]`. **Lesson:** multi-artifact stories (runbook + workflow YAML + JSON config + Node.js script + regression guard test) warrant at least two review rounds — the first round focuses on the runbook and tests, the second on the workflow/script robustness the first round under-weighted.

7. **`project-context.md` is a load-bearing artifact for autonomous dev agents — but its rules must be back-applied to prior work when adopted mid-epic.** The `--fail`/`--max-time` rule was codified mid-Epic-4 (after Story 4.9). Stories 4.10 and 4.11 applied it from creation. Stories 4.7, 4.8, and 4.9 did not — and each non-compliance is now a deferred finding. **Lesson:** when a project-context rule is adopted mid-epic, the adopting story should include a "back-apply to prior work" task — OR explicitly decide to leave prior work non-compliant and record the decision (not defer the finding).

8. **Epic 4 ran in parallel with Epic 5 (UX visual drift) — and the parallel execution created invisible coordination costs.** The git log shows Epic 4 and Epic 5 commits interleaved (Epic 5's Story 5.5 architectural pivot happened between Epic 4's Stories 4.8 and 4.9). Story 4.7's working tree contained an unrelated n8n workflow change likely from another story's branch. The two epics shared the `feat/epic-4` branch at points. **Lesson:** parallel epics need an explicit coordination protocol — either (a) separate long-lived branches with a merge cadence, or (b) interleaved stories with explicit working-tree-cleanup verification between stories. The current "both epics on `main`" practice risks cross-epic contamination.

## Key Insights from this Retro

1. **A "deferred as human action items" deferral is structurally worse than "deferred to a future story" — it floats between stories with no single owner.** The Story 4.5 → 4.6 → 4.7 cascade is the canonical anti-pattern. Codify as a team agreement.
2. **Team agreements without a backing automation hook are not operationalized.** The Epic 5 retro reaffirmed Wave-1+2 closeout at every natural closeout boundary; Epic 4 did not apply it. The retro发现了 it after the fact.

4. **DP-2 amendments (spec vs code contradiction) recurred 4 times — each correctly resolved, but the pattern suggests spec authoring is not verifying spec claims against current code at story-creation time.** Add a "verify spec claims against current code" step to story creation.
5. **Epic 4 ran in parallel with Epic 5 — the parallel execution created invisible coordination costs** (working-tree contamination risk, branch interleaving). Parallel epics need an explicit coordination protocol.
6. **Story 4.8's strict zero-deferral policy (26 patches inline, 0 deferred) is the new baseline** for stories with explicitly bounded scope ("documents recovery, not deploy-mechanism fixes"). When there are no DP-5 scope-temptation tradeoffs to make, the deferral pressure disappears.
