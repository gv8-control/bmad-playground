Poor

This PRD is structurally strong: all seven rubric dimensions score strong or adequate, the Vision has a specific thesis, every FR carries testable consequences, NFRs have real numeric targets, and scope decisions are named honestly rather than buried. A reviewer sympathetic to the product would call this a well-made planning document.

The adversarial review shifts the grade to Poor by surfacing three critical strategic gaps the rubric cannot reach. The PAT onboarding step directly contradicts the access-barrier thesis the product is built on, and the success metric (SM-1) will return signal about infrastructure readiness before it returns signal about product-market fit. The competitive window argument claims durable advantages over Claude Code Web that Anthropic could reproduce within the stated window, with no moat analysis to close that gap. The cost model omits Daytona sandbox compute entirely, leaving the $25–$30/seat price point unvalidated against the real cost floor. These are not PRD writing problems — they are unresolved strategic bets the PRD presents as settled, and a skeptical decision-maker at a green-light gate would challenge all three.

Decision-readiness

Strong

Trade-offs are surfaced explicitly and not papered over. The PAT onboarding friction is named as "a known activation friction point" with a documented post-MVP trigger; the main-branch-only constraint is called "a conscious scope decision" with last-write-wins acknowledged as a known failure mode; agent-initiated commit failure is explicitly classified as "a known silent failure mode in MVP." These are real decisions, stated as decisions.

Open questions are honest: Q-1 acknowledges an unvalidated NFR-P2 boundary and assigns the action to the architect before the architecture document is finalized. The NOTE FOR PM callout in §6.2 on GitHub App integration gives concrete guidance (fast-track based on trial activation data) rather than a safe hedge.

One thin spot: the single-container backend constraint (§8) is stated plainly, but the blast radius — no burst handling, single point of failure at launch — is not named as a decision with a specific consequence if usage spikes.

Low

Missing blast radius on single-container constraint

§8
The "conscious scope decision" label is correct but the consequence (no burst handling, single point of failure at launch) is not named.
Fix: Add one sentence naming the consequence: e.g., "If burst usage at launch saturates the container, new Conversation opens will queue or fail; the documented mitigation is monitoring-triggered vertical scale."

Strong

No copied boilerplate found.

Strategic coherence

Strong

The PRD has a clear thesis: BMAD's access layer is broken for non-devs; bmad-easy fixes the access layer without changing the methodology. Every scoped feature serves this directly. Feature prioritization follows from the thesis — branching/PR workflows are post-MVP because they serve developers, not the non-dev access problem. Success metrics validate the thesis correctly: SM-1 (unassisted session completion) is the thesis's testable consequence; counter-metrics SM-C1 and SM-C2 prevent optimizing for session duration and seat growth, which would contradict the thesis.

One gap: SM-4 is missing from the sequence (SM-1, 2, 3, 5, 6). Whether this is a renaming artifact or a deleted metric is unclear.

Medium

SM-4 missing from Success Metrics sequence

§11
The sequence is SM-1, SM-2, SM-3, SM-5, SM-6 — no SM-4. If intentionally removed, remaining metrics should be renumbered. If accidentally omitted, the gap signals a missing validation signal.
Fix: Either renumber SM-5 and SM-6 to SM-4 and SM-5, or document what SM-4 was and confirm it was intentionally removed.

Done-ness clarity

Adequate

The majority of FRs carry testable consequences with specific, verifiable conditions. FR-1 specifies exact error conditions on validation failure; FR-10 gives the exact latency target (1,500 ms first token), exact interaction model, and exact scroll behavior; FR-15 specifies the exact commit message format, no-op behavior on a clean working tree, and queuing behavior during an agent turn.

Three FRs partially defer done-ness: FR-16 delegates Artifact ordering to the UX spec; FR-3 specifies that commits are attributed to "the user's platform identity (name and email)" without naming the data source for the email; FR-18 defers session lifetime with an explicit NON-GOAL callout, but "whatever the framework default provides" spans 15 minutes to indefinite.

Adversarial review

review-adversarial-general.md

The PRD is unusually self-aware — it names its own assumptions, marks friction points, and avoids most aspirational language. The adversarial lens surfaces a different class of problem: the product's core thesis rests on a target user doing something that developers find routine (PAT generation, slash-command invocation), the competitive window argument contains a contradiction it never resolves, the cost model is optimistic in ways that could invert unit economics, and several NFRs are either untestable or will fail under realistic conditions.

Critical

PAT onboarding contradicts the product's access-barrier thesis

§4.1 FR-1, §6.2, §9
The platform's stated purpose is to give non-dev PMs browser-based access without touching the terminal. The first required action is generating a fine-grained GitHub PAT with contents:write scope — navigating GitHub developer settings, understanding permission scopes, pasting a long token string. The post-MVP trigger ("if identified as a material activation blocker in beta") is circular: you cannot measure PAT friction as a blocker until after users have bounced on it, at which point SM-1 signal is already compromised.
Fix: Either provide a usability test result showing ≥N% of PM-personas completing PAT setup unassisted, or state that SM-1's 60% target is contingent on PAT setup succeeding, with a separate funnel metric tracking PAT drop-off so the two signals are not conflated.
Critical

Competitive window argument is self-contradictory

§9
§9 lists bmad-easy's "durable advantages" over Claude Code Web as team billing, BMAD-structured sessions, automatic Artifact commitment, Project Map, no per-user subscription required. Each advantage is either trivially reproducible by Anthropic (team billing, session structuring via system prompts) or dependent on BMAD's continued relevance (BMAD Skills). The window is stated as 12–24 months in one sentence and "4–10 months remaining" in another — a range of 4–16 months with no committed number. No evidence is provided that Anthropic's roadmap excludes native team billing or workflow templates.
Fix: Distinguish bmad-easy's advantage as structural (BMAD methodology lock-in, community distribution, existing user base) rather than feature parity. State explicitly what the durable moat is if Anthropic ships comparable features within the window.
Critical

Cost model omits Daytona compute; unit economics unvalidated

§10
The cost model estimates $3–$6 LLM cost per active seat per month against a $25–$30 price point, implying 75–85% gross margin. Daytona sandbox compute is omitted — noted as "add[ing] cost on top" with no estimate. At SM-6's target of ≥4 sessions per month per active seat, sandbox compute is non-trivial. The weighted average of $0.77/session also assumes prompt caching offsets that depend on session structure and cache TTL. "Validate before launch pricing is locked" has no owner and no date.
Fix: Add an explicit Daytona compute cost estimate (even a range) in §10. State a gross margin floor below which the pricing model must change. Name an owner and a date for the pre-launch cost validation.
High

NFR-P2 is a smoke test, not a verified requirement

§7 Performance, §12 Q-1
"Verified with a single manual test run under normal conditions" is a smoke test. NFR-P2 does not define: what "normal conditions" means, what repository size was used, what Daytona cold start latency looks like at p90, or what happens under provisioning queue contention. §12 Q-1 acknowledges the large-repository failure mode but the failure is not limited to large repos — a fresh Daytona sandbox provisioning, cloning, and starting Claude Code in under 10 seconds is aggressive even for a 20 MB repository at p90.
Fix: Source a measured Daytona cold start baseline (p50, p90) from the technical research. State the maximum repository size for which NFR-P2 is guaranteed. Add a commitment to surfacing a degraded-mode message when the target is missed rather than silently delaying.
High

Silent agent-initiated commit failure is first-session churn risk

§5, FR-12, FR-14
§5 explicitly states: "If the Agent's git commit fails during a Conversation, no error is surfaced to the user and no retry is attempted." A user who completes a 20-minute PRD session may see "Progress saved" appear (Semantic Pill fires when the Agent attempts a commit) and have no Artifact in the repository if the commit failed — with no notification. FR-14's Working Tree State Indicator does not update on agent-commit failure; it tracks working tree state, not commit success. A PM who loses 20 minutes of work with no feedback will not return.
Fix: Either surface a visible error when an agent-initiated commit fails in MVP, or explicitly state in FR-12 that Semantic Pills for "Progress saved" are emitted only on confirmed commit success — and define how confirmation is detected before the pill fires.
High

Project Map "In Progress" is misleadingly scoped

§4.2 FR-5, FR-6
The Project Map claims to show "what BMAD work the team has done and what is in progress." But "In Progress" status is derived from active platform Conversations only. A developer running a session locally will never appear as in-progress. FR-6 states "Artifacts produced by developers through local tooling appear alongside platform-produced Artifacts; the source of production is not distinguished" — but locally in-progress work is invisible entirely, not just undistinguished. A PM looking at the Project Map during a developer's active local session sees no in-progress indicator for that work.
Fix: Clarify in FR-6's consequences that "In Progress" status is limited to active platform Conversations. Update the §4.2 description to reflect that locally-run agent sessions are not visible in any status, so the "team view" claim is scoped accurately.
High

Commit attribution mechanism is unspecified and may fail silently

§4.1 FR-3
FR-3 states "the PAT is used for HTTPS transport only; it does not appear in the git commit record" — achievable by injecting git user.name and user.email into the Sandbox's git config. But no mechanism is described. If the Agent runs git commit inside the Sandbox with a default git config, the commit author will be whatever the Sandbox contains (likely empty or a platform default). GitHub OAuth primary email can be private, returning a noreply address. The requirement is real but implementation-free.
Fix: Specify the source for attribution identity (e.g., "GitHub OAuth primary email, or user-provided override in Settings"). Note that git config injection is a required step in Sandbox initialization. Address the GitHub private email case explicitly.
Medium

EU Data Act claim has no derived requirements

§8
§8 states "EU Data Act… Data portability in machine-readable formats and mandatory switching rights must be designed into the product architecture from launch. These cannot be retrofitted." No requirements in §4 implement this. No data export format, switching mechanism, or implementation scope is defined. "These cannot be retrofitted" is correct but adding a compliance assertion with zero derived requirements is risk displacement, not compliance planning.
Fix: Either derive concrete EU Data Act requirements (at minimum: what "machine-readable export" means for Artifacts, what the switching mechanism is) or explicitly defer them to post-MVP with a named risk owner and an acknowledgment that this creates launch risk for EU customers.
Medium

Single-container restart impact on active Conversations is unaddressed

§8
A container restart (deploy, crash, OOM) terminates all active Conversations and Sandboxes without notice. FR-13 guarantees chat history is recoverable, but in-progress working tree state is not (NFR-R2). For a user mid-session, this is indistinguishable from a product failure. Expected restart frequency during MVP is not stated; user-visible recovery path is not defined.
Fix: State expected restart frequency during MVP (e.g., "deploys will cause downtime N times per week"). Add a user-visible recovery path for interrupted sessions (e.g., "Session interrupted — your committed work is safe; click to resume").
Low

Three-hop buyer journey has no platform-assisted conversion mechanism in MVP

§2.1
The GTM model is: developer discovers BMAD → finds bmad-easy → builds internal case → economic buyer approves purchase. MVP has no trial urgency (all users auto-enrolled with no expiry), no conversion event, and no tool to help the developer build the internal case. The developer cannot invite teammates during trial (seat management is post-MVP), so the bottleneck the developer is motivated to resolve does not get resolved during evaluation.
Fix: State an early-access onboarding model that lets the developer champion add one or two non-dev teammates during trial — even informally. Without this, the purchase path is aspirational and SM-5 (VP/Director buyer conversion) has no observable mechanism.
