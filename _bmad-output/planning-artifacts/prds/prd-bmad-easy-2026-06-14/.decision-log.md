## 2026-06-14 — UX review: chat interface gap analysis

**Context:** UX Designer audited the Skill Session chat tab (FR-9 through FR-13) against the baseline of modern LLM chat interfaces and identified 12 gaps. The PM selected 8 points plus one additional item (cold-start spinner) for immediate inclusion. Points not actioned are noted below.

**DL-001 — Stop button behavior (pt.1)**
Stop button terminates the in-flight LLM response and any tool or Bash process running inside the Sandbox without terminating the Sandbox itself. Rationale: destroying the Sandbox on Stop would force a cold-start on every correction — unacceptable for sessions that may run 20+ minutes. Reference: anthropics/claude-code#3455. Implementation note (addendum): the platform must send an abort signal (e.g. SIGINT) to the running Claude Code process tree without triggering a Daytona Sandbox shutdown; the backend must distinguish a user-initiated Stop from an idle timeout. Placed in FR-10.

**DL-002 — Copy-to-clipboard (pt.3)**
Per-message copy action (hover-accessible) and per-code-block copy button both required. Rationale: agents produce long structured outputs mid-session; non-dev users need to paste fragments into Slack, meeting notes, and emails without selecting text manually. Placed in FR-10.

**DL-003 — Scroll-to-bottom button (pt.4)**
Button appears when user scrolls above the most recent message; hidden at bottom. Auto-scroll continues while streaming unless user has manually scrolled up. Rationale: sessions run 20+ minutes; scroll position management is table-stakes for any streaming chat interface. Placed in FR-10.

**DL-004 — Multi-line auto-growing textarea (pt.5)**
Chat input is a multi-line auto-growing textarea, not a single-line `<input>`. Rationale: PMs providing rich project context will write multi-paragraph inputs; a single-line field creates avoidable friction. Resolves ambiguity in the prior "plain text field" language. Placed in FR-10.

**DL-005 — Keyboard submit behavior (pt.6)**
Enter submits; Shift+Enter inserts a newline; Send button present as secondary affordance. Rationale: de facto standard across all major LLM chat interfaces; deviation confuses non-dev users. Placed in FR-10.

**DL-006 — Message timestamps (pt.7)**
Each message displays a timestamp: relative ("just now") for recent messages, wall-clock time for older ones. User timestamps hover-accessible; agent timestamps shown inline at reduced prominence. Rationale: sessions are reviewed after the fact, especially after sandbox expiry. Placed in FR-10.

**DL-007 — Browser tab titles (pt.10)**
Skill Session tab titles reflect the Skill name, appended with " (expired)" when the session ends. Project Map tab title reflects the project name. Rationale: undifferentiated tab titles make multi-tab navigation impossible; required for the concurrent-session workflow in FR-11. Placed in FR-11.

**DL-008 — Input draft persistence (pt.11)**
Unsent drafts persisted to browser `localStorage` per Skill Session tab. Restored on reconnect within idle timeout. Cleared on send. Rationale: accidental refresh during long message composition is a common non-dev frustration; low-cost fix. Placed in FR-10.

**DL-009 — Cold-start spinner in chat area**
While the Sandbox provisions, a spinner and "Starting session…" label are displayed in the chat area; input is disabled until ready. Rationale: original FR-9 only said "loading indicator" without specifying placement; chat-area placement sets correct spatial expectation and prevents typing into a non-ready session. Placed in FR-9.

## 2026-06-14 — UX review: navigation model and onboarding decisions

**Context:** UX Designer (Sally) reviewed the PRD, identified gaps, and the PM resolved them. Changes applied to PRD in this session.

**DL-010 — PAT onboarding simplified**
In-product step-by-step guide replaced with a link to GitHub documentation + a single PAT input field. Rationale: the guide was over-specified in the PRD; GitHub docs are the authoritative and maintained source for PAT generation steps. Updated §4.1 description, FR-1 first consequence, UJ-1 path.

**DL-011 — Navigation model: page-based with breadcrumbs**
The platform uses page-based navigation — no in-app tab UI. Project Map, Skill Session, and Artifact Browser are distinct pages with stable URLs. In-page navigation is hierarchical with breadcrumbs one level deep (Project Map is root; session and Artifact Browser pages show a breadcrumb back). Rationale: avoids cross-tab browser API complexity while preserving the ability for users to naturally open multiple sessions in browser tabs. Added to §8 Constraints.

**DL-012 — Tab language replaced with page-based language throughout**
All "tab" references (FR-9, FR-11, §4.3 description, §6.1, UJ-2) updated to reflect page-based model. FR-11 renamed from "Multi-Session Tabs" to "Concurrent Skill Sessions". FR-8 "brings tab into focus" → "navigates to that session's page". FR-11 removed "Project Map tab is always present" (now just the home page). Session limit enforcement remains server-side.

**DL-013 — Slash-command Skill entry**
Skill Session initiation uses slash-command UI: typing `/` in the Project Map's skill entry field displays a filterable list of matching Skills; selecting one navigates to a new session page. Rationale: matches Claude Code's familiar interaction model; eliminates the need for a separate Skill Selection UI surface. Updated FR-9 consequences, §4.3 description, UJ-2 path, §6.1.

**DL-014 — Tool call visibility added to FR-10**
A tool execution indicator is shown during Bash/tool execution inside the Sandbox, distinct from the thinking indicator for LLM response generation. Rationale: BMAD agents execute tools that can take 15–30 seconds; a silent UI during this window would feel broken to non-dev users. Added to FR-10.
