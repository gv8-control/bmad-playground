bmad-easy — UX Aesthetics Validation Report

_bmad-output/planning-artifacts/ux-designs/ux-bmad-easy-2026-06-15/

The aesthetic direction is sound: a coherent blue-violet dark palette, elevation via surface lightness (no shadows), a restrained violet accent reserved for primary actions, and a clear two-typeface typographic ramp. Execution, however, has real gaps that a downstream builder would propagate.

A broken color-token reference ({colors.warning} / {colors.warning-bg} — defined nowhere in the frontmatter) was introduced when the Access Notice component was added on 2026-07-02 and was not caught by the prior rubric pass, which ran the same day. The Access Notice also lacks a components: frontmatter entry — it has prose only. The muted text token (text-3, #56556A) fails WCAG AA contrast on every surface (~2.3–2.7:1; AA requires 4.5:1) and is used for permanent informational text (timestamps, "All saved" indicator). The stated 4px spacing grid is violated throughout the mockups and in the spine's own component specs (7px, 10px, 11px, 14px appear routinely; the credential-error-banner spec itself uses 10px). Cross-mock consistency is uneven: the conversation mock diverges from the other four app-shell mocks in nav-item display model, textarea text color, and send-button class naming.

None of this is fatal to the aesthetic vision. The palette, elevation system, typography, and shape language are well-conceived. But the three critical findings (broken tokens, inaccessible text, missing component spec) and the six high-severity findings should be resolved before a builder mirrors these artifacts.

Colors

thin

Palette coherence is strong — a blue-violet dark theme with restrained accent use and clear semantic-color discipline. However, a broken token reference, WCAG AA contrast failures on the muted text token, inconsistent primary-action treatment, an invisible border token, ad-hoc colors in mocks, and inconsistent focus-ring/loading-state implementations bring the verdict down to thin.

critical

Broken color-token references: {colors.warning} / {colors.warning-bg} undefined

DESIGN.md line 381, EXPERIENCE.md line 262
The Access Notice component prose references {colors.warning-bg} and {colors.warning} for its background and left border, but the frontmatter colors: block defines no warning or warning-bg token. The semantic palette has positive/caution/negative only. Downstream code mirroring the spine cannot resolve these references. The prior rubric review (§2, "strong") did not catch this because the Access Notice was added as part of applying that review's own findings on the same day.
Fix: Add warning / warning-bg tokens to the frontmatter, or replace with {colors.caution} / {colors.caution-bg} (the closest existing semantic).
critical

text-3 (#56556A) fails WCAG AA contrast on every surface

DESIGN.md line 18
Computed ratios: 2.68:1 on bg, 2.49:1 on surface, 2.29:1 on surface-raised. AA requires 4.5:1 for normal text. text-3 is used for permanent informational text: agent message timestamps (shown permanently per DESIGN.md line 337), the "All saved" clean-state indicator (line 352), footer links, disabled text. No contrast targets are stated anywhere in DESIGN.md.
Fix: Lighten text-3 to meet 4.5:1 on all surfaces (~#757488 or lighter), or restrict text-3 to placeholder/disabled use and move timestamps and "All saved" to text-2. State contrast targets in DESIGN.md §Colors.
high

Sign-in primary button uses text-1 (#EDECF5) as background instead of accent (#7B6EE8)

key-signin.html line 131
DESIGN.md Do's: "Use {colors.accent} for all primary interactive elements." The onboarding "Connect repository" button in the same pre-app-shell context uses accent. Two primary-action treatments with no documented rationale.
Fix: Use accent for the sign-in button, or document the exception in DESIGN.md.
high

border-subtle (#1E1E26) equals surface-raised (#1E1E26) — invisible on raised surfaces

DESIGN.md lines 11, 13
border-subtle is for "inner divisions on raised surfaces" but is the same color as the raised surface it sits on. Visible only against surface (#16161C), as in the side-nav right border.
Fix: Set border-subtle to a value between surface and border (e.g., #232330).
medium

Ad-hoc colors not in the palette

key-project-map.html line 218, key-new-conversation.html line 351, key-artifact-browser.html line 209
Artifact-card hover border #3b3b4f (no token). Slash-picker item hover bg #2B2B38 (border token used as bg; spine says surface-raised). List-entry hover rgba(30,30,38,0.6) (ad-hoc semi-transparent surface-raised).
Fix: Replace with nearest token; add a hover-surface token if surface-raised is too subtle.
medium

Focus-ring implementation inconsistent and non-compliant with spine

key-onboarding.html line 142, key-new-conversation.html line 264
Spine specifies "2px outline, 2px offset" in accent. Onboarding input uses 20% opacity box-shadow; chat input uses 15% opacity. Different opacities; neither is a 2px outline with 2px offset. Project-context.md canonical: focus:ring-2 focus:ring-accent focus:ring-offset-2.
Fix: Standardize on the spine's 2px solid accent outline with 2px offset across all mocks.
medium

Loading/disabled button states use ad-hoc colors with no spine tokens

key-onboarding.html line 196, key-new-conversation.html lines 310–318
connect-button.loading uses #56556A (text-3) as bg. send-button.disabled uses #1E1E26 (surface-raised) bg + #56556A text. send-button.loading uses #56556A bg. Two different disabled/loading treatments; neither tokenized.
Fix: Add disabled/loading button-state tokens, or specify as "accent at 40% opacity" with a documented value.
low

Browser-chrome lock icon uses positive (#3ECF8E) decoratively

all 7 mocks, e.g. key-signin.html line 51
DESIGN.md Don'ts: "Never use them [semantic colors] decoratively." Mock furniture only, not product UI.
Fix: Use a neutral color for mock chrome, or note that browser-chrome furniture is outside the design system scope.
low

Spinner treatments inconsistent between mocks

key-onboarding.html lines 203–205, key-new-conversation.html lines 225–226
Onboarding spinner: white track at 30% opacity, white head. New-conversation spinner: border (#2B2B38) track, accent (#7B6EE8) head. Different visual language for the same loading concept.
Fix: Standardize on one spinner treatment; document in DESIGN.md.

Margins & Spacing

adequate

The 4px base unit and Tailwind scale are clearly stated, and many values comply. But the grid rule is widely violated — in the mockups and in the spine's own component specs — and the spacing scale is not tokenized in the frontmatter. Cross-surface padding inconsistencies and undocumented max-widths round out the picture. The direction is right; the discipline is missing.

high

4px grid violated throughout mockups and spine specs

DESIGN.md line 266, frontmatter line 186; all app-shell mocks
Off-grid values: 7px (nav-item padding, all 5 app-shell mocks), 10px (credential-error-banner in DESIGN.md frontmatter line 186; list-entry; teaser-item; field-group gap), 11px (GitHub button; connect-button), 14px (field-input; teaser-item), 6px (frontmatter-badge; nav active margin). The 10px in the spine's own frontmatter is notable — the grid rule is violated inside the spine itself.
Fix: Snap all off-grid values to nearest 4px multiple (7→8, 10→8 or 12, 11→12, 14→12 or 16, 6→4 or 8). Document 2px half-step exception if needed for tight controls.
medium

spacing frontmatter has only base: 4px — no scale tokens

DESIGN.md lines 61–63
Tailwind scale (4, 8, 12, 16, 20, 24, 32, 40, 48, 64) is prose-only. Downstream code can't reference {spacing.4} etc. Contributes to off-grid proliferation.
Fix: Add scale tokens ('1': 4px, '2': 8px, …) to the spacing frontmatter.
medium

Chat input area bottom padding inconsistent between chat surfaces

key-new-conversation.html line 245, key-conversation.html line 322
New-conversation: padding: 0 32px 24px. Conversation: padding: 0 32px 20px. 24px vs 20px for the same component on two surfaces.
Fix: Standardize on one value; document in the chat-input component spec.
medium

Undocumented max-widths across mocks with no system

key-signin.html line 65, key-onboarding.html line 65, key-project-map.html line 216, key-artifact-browser.html line 301, key-conversation.html line 222
Auth card 400px, onboarding card 480px, artifact card 720px, md-content 720px, message row 824px (760 + 64 padding). The 760px chat max-width is documented; the others are not. 400, 480, 720, 760, 824 have no documented relationship.
Fix: Document a content-width system in DESIGN.md §Layout & Spacing, or note as per-surface decisions.
low

Tool-pill padding (2px 8px) vs semantic-pill padding (4px 10px) — different sizing

DESIGN.md frontmatter lines 97, 106
Semantic pill is described as "elevated variant of Tool Pill" but has different padding. When both appear in the same message stream, they render at different heights. 2px is also off the 4px grid.
Fix: Align semantic-pill padding to match tool-pill, or document the size relationship if intentional.
low

Status-badge padding (2px 8px) differs from semantic-pill padding (4px 10px)

DESIGN.md frontmatter lines 141, 106
Both are pill-shaped badges with inconsistent internal sizing.
Fix: Standardize pill-badge padding across tool-pill, semantic-pill, and status-badge variants.

Consistency

adequate

The spines are internally well-structured — section ordering, token cross-references (except the broken warning tokens), and component naming are largely consistent. But the mockups diverge from the spines and from each other in several places: nav-item display model, textarea text color, send-button class naming, active-state margins, shadow discipline, scrollbar behavior, and wordmark presentation. The Access Notice component is the most serious consistency gap — prose without a frontmatter spec, referencing undefined tokens.

critical

access-notice component has prose but no frontmatter spec

DESIGN.md lines 380–381
The components: frontmatter defines 16 components; the Access Notice is described in §Components prose but has no access-notice key. Its prose references non-existent {colors.warning} / {colors.warning-bg} tokens. A downstream builder has no machine-readable spec.
Fix: Add an access-notice: entry to the frontmatter with resolved token references, or fold into credential-error-banner as a variant.
high

Nav-item display property inconsistent across mocks

key-conversation.html line 114 vs others line 103
Conversation mock uses display: flex; all other app-shell mocks use display: block. Changes vertical alignment and hit-area of nav items.
Fix: Standardize on one display model across all app-shell mocks.
high

Conversation mock textarea uses text-3 (#56556A) for entered text instead of text-1

key-conversation.html line 344
Entered text renders at 2.29:1 contrast — nearly invisible. The new-conversation mock correctly uses #EDECF5 (text-1). If a builder mirrors the mock, typed text would be unreadable.
Fix: Set textarea color to #EDECF5 (text-1); reserve text-3 for ::placeholder only.
high

Send-button class naming inconsistent

key-new-conversation.html line 294 vs key-conversation.html line 360
.send-button vs .send-btn — different class names for the same component across mocks.
Fix: Standardize on one class name.
medium

Nav-item active-state margins inconsistent across mocks

key-project-map.html line 117, key-conversation.html lines 131/138, key-settings.html line 138
1px vs 0px top/bottom margin creates subtle vertical position shifts. The conversation mock has two active classes (.active and .active-conv) with different margins.
Fix: Standardize active-state margins; use one active class.
medium

Slash picker uses box-shadow, contradicting the no-shadow rule

key-new-conversation.html line 332
DESIGN.md Don'ts: "Don't use box shadows to communicate elevation on dark surfaces." The slash picker is a floating dropdown that uses shadow for visual separation. No floating-element exception is documented.
Fix: Document a floating-element exception in §Elevation & Depth, or replace shadow with stronger border/contrast.
medium

Scrollbars hidden in multiple mocks, undocumented in spine

key-project-map.html line 204, key-conversation.html line 217, key-artifact-browser.html lines 199/261
::-webkit-scrollbar { display: none; } in 4 mocks. Spine doesn't mention scrollbar styling. Hidden scrollbars harm usability and accessibility.
Fix: Document the decision in EXPERIENCE.md, or show styled scrollbars.
medium

Wordmark presentation inconsistent between pre-app-shell and app-shell

key-signin.html lines 231–233 vs key-project-map.html line 282
Sign-in: logo box ("be" in accent-filled 12px square) + "bmad-easy" name + tagline. App-shell: text "bmad·easy" with middle dot in accent. Different brand presentations with no documented relationship.
Fix: Document both wordmark treatments in DESIGN.md §Brand & Style, or standardize.
low

Dead CSS: .annotation-band defined but unused in 4 mocks

key-project-map.html, key-new-conversation.html, key-conversation.html, key-artifact-browser.html
CSS block defined but not rendered in the HTML body. Could confuse a builder reading the mock source.
Fix: Remove unused CSS blocks.
low

min-width: 1280px on body doesn't match spine's 1024px minimum

all 7 mocks; EXPERIENCE.md line 16
Mocks don't demonstrate the 1024px minimum or the tablet (768–1023px) collapsed-nav behavior. A builder wouldn't know what the UI looks like at 1024px.
Fix: Render mocks at 1024px, or note that mocks are at 1280px for readability and the 1024px layout is spine-only.

Mechanical notes

  • Broken cross-reference (critical): {colors.warning} and {colors.warning-bg} referenced in DESIGN.md line 381 and EXPERIENCE.md line 262 — neither exists in the frontmatter. Prior rubric review (§2, "strong") did not include these in its checked-token list.
  • Missing component frontmatter (critical): access-notice described in DESIGN.md §Components prose (line 380) but absent from the components: frontmatter (16 entries; should be 17).
  • Contrast targets unstated: DESIGN.md §Colors does not state WCAG contrast targets for any text/surface combination. text-3 fails AA everywhere; accent as text on surface is borderline (4.50:1).
  • Spacing scale not tokenized: spacing frontmatter has base: 4px only; Tailwind scale is prose-only. 6 distinct off-grid values (6, 7, 10, 11, 14, 28) appear across mocks and spine specs.
  • Class naming drift: .send-button vs .send-btn across two mocks for the same component.
