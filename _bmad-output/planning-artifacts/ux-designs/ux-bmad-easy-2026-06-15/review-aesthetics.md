# Aesthetics Review — bmad-easy

## Scope

Focused aesthetics review of the spine pair (`DESIGN.md` + `EXPERIENCE.md`, both `status: final`, updated 2026-07-04) and all 7 key-screen mockups in `mockups/`. Three lenses: **colors**, **margins/spacing**, **consistency**. Prior reviews (`review-rubric.md` 2026-07-02, `review-implementation-drift.md` 2026-07-05) covered contract coverage and implementation drift — neither examined aesthetics. This review fills that gap.

Contrast ratios computed per WCAG 2.1 relative-luminance formula. Spine line citations refer to the current `DESIGN.md` / `EXPERIENCE.md`.

## Overall verdict

The aesthetic direction is sound: a coherent blue-violet dark palette, elevation via surface lightness (no shadows), a restrained violet accent reserved for primary actions, and a clear typographic ramp. The execution, however, has real gaps. A broken color-token reference (`{colors.warning}` / `{colors.warning-bg}` — defined nowhere in the frontmatter) was introduced when the Access Notice component was added on 2026-07-02 and was not caught by the prior rubric pass. The muted text token (`text-3`, `#56556A`) fails WCAG AA contrast everywhere it is used for informational text (~2.3–2.7:1 against every surface; AA requires 4.5:1). The stated 4px spacing grid is violated throughout the mockups and even in the spine's own component specs (7px, 10px, 11px, 14px appear routinely). Cross-mock consistency is uneven: the conversation mock diverges from the other four app-shell mocks in nav-item display model, textarea text color, and button class naming. None of this is fatal to the aesthetic vision, but a downstream builder mirroring these artifacts would propagate broken tokens, inaccessible text, off-grid spacing, and cross-surface inconsistencies.

## 1. Colors — thin

Checked: palette coherence, token resolution, contrast ratios for all load-bearing text/background combinations, semantic-color discipline, accent usage rules, focus-ring color, loading/disabled state colors, ad-hoc values.

### Findings

- **critical** Broken color-token references: `{colors.warning}` and `{colors.warning-bg}` are undefined (DESIGN.md line 381, EXPERIENCE.md line 262). The Access Notice component prose references `{colors.warning-bg}` and `{colors.warning}` for its background and left border, but the frontmatter `colors:` block (lines 7–34) defines no `warning` or `warning-bg` token. The semantic palette has `positive`/`caution`/`negative` only. Downstream code mirroring the spine cannot resolve these references. The prior rubric review (§2, "strong") did not catch this because the Access Notice was added as part of applying that review's own findings on the same day. *Fix:* either add `warning` / `warning-bg` tokens to the frontmatter (if a distinct warning semantic is intended, distinct from `caution`), or replace `{colors.warning}` / `{colors.warning-bg}` with `{colors.caution}` / `{colors.caution-bg}` (the closest existing semantic — "something needs attention" — which is what an Access Notice communicates).

- **critical** `text-3` (`#56556A`) fails WCAG AA contrast on every surface (DESIGN.md line 18). Computed ratios: 2.68:1 on `bg` (#0D0D11), 2.49:1 on `surface` (#16161C), 2.29:1 on `surface-raised` (#1E1E26). AA requires 4.5:1 for normal text, 3:1 for large text. `text-3` is used for permanent informational text: agent message timestamps (DESIGN.md line 337 — "shown permanently but at low prominence"), the "All saved" clean-state working-tree indicator (DESIGN.md line 352), footer links (key-signin.html line 178), and disabled text. Placeholders are exempt from WCAG, but timestamps and the "All saved" indicator are not — they convey state the user must read. No contrast targets are stated anywhere in DESIGN.md for any color combination (the rubric requires this for load-bearing combinations). *Fix:* either lighten `text-3` to meet 4.5:1 on all three surfaces (approximately `#757488` or lighter), or restrict `text-3` to placeholder/disabled use only and move timestamps and the "All saved" indicator to `text-2` (#8D8CA0, which passes at ~5.0:1 on surface). State contrast targets in DESIGN.md §Colors for text-1/text-2/text-3 against bg/surface/surface-raised.

- **high** Sign-in primary button uses `text-1` (#EDECF5) as background instead of `accent` (#7B6EE8) (key-signin.html line 131). DESIGN.md Do's: "Use `{colors.accent}` for all primary interactive elements — buttons, focus rings, active states, links" (line 392). The GitHub OAuth button is the sole primary action on the sign-in page. The onboarding "Connect repository" button — the same pre-app-shell context, same action priority — uses `accent` (#7B6EE8) as its background (key-onboarding.html line 180). Two primary-action treatments on two pre-app-shell surfaces with no documented rationale for the difference. *Fix:* either use `accent` for the sign-in button (consistent with onboarding and the spine's Do's), or document the exception in DESIGN.md if a high-contrast OAuth convention is intentionally overriding the accent rule.

- **high** `border-subtle` (#1E1E26) is the same value as `surface-raised` (#1E1E26) (DESIGN.md lines 11, 13). DESIGN.md says `border-subtle` is for "inner divisions on raised surfaces" (line 13). A border the same color as the surface it sits on is invisible. `border-subtle` is visible only when placed against `surface` (#16161C) — e.g., the side-nav right border (DESIGN.md line 69). On any raised surface (inputs, cards, dropdowns), an inner divider using `border-subtle` disappears. *Fix:* set `border-subtle` to a value between `surface` (#16161C) and `border` (#2B2B38) — e.g., `#232330` — so it is visible on both `surface` and `surface-raised`.

- **medium** Ad-hoc colors not in the palette appear in mockups. key-project-map.html line 218: artifact-card hover border `#3b3b4f` (no token). key-new-conversation.html line 351: slash-picker item hover background `#2B2B38` (the `border` token used as a background; the spine says hovered items use `{colors.surface-raised}` #1E1E26 — DESIGN.md line 361). key-artifact-browser.html line 209: list-entry hover `rgba(30,30,38,0.6)` (ad-hoc semi-transparent `surface-raised`). *Fix:* replace ad-hoc values with the nearest token; add a `hover` surface token if the existing `surface-raised` is too subtle for hover affordance.

- **medium** Focus-ring implementation inconsistent and non-compliant with the spine (key-onboarding.html line 142, key-new-conversation.html line 264). The spine specifies "2px outline, 2px offset" in `{colors.accent}` (EXPERIENCE.md line 378). The onboarding input uses `box-shadow: 0 0 0 2px rgba(123,110,232,0.2)` (20% opacity). The chat input uses `box-shadow: 0 0 0 2px rgba(123,110,232,0.15)` (15% opacity). Different opacities for the same treatment; neither is a 2px outline with 2px offset. The project-context.md canonical implementation is `focus:ring-2 focus:ring-accent focus:ring-offset-2 focus:ring-offset-surface` (solid ring, not translucent). *Fix:* standardize on the spine's 2px solid accent outline with 2px offset across all mocks; pick one opacity if a translucent ring is preferred and document it.

- **medium** Loading/disabled button states use ad-hoc colors with no spine tokens (key-onboarding.html line 196, key-new-conversation.html lines 310–318). `connect-button.loading` uses `#56556A` (text-3) as background. `send-button.loading` uses `#56556A`. `send-button.disabled` uses `#1E1E26` (surface-raised) as background with `#56556A` text. Two different disabled/loading treatments for buttons; neither is tokenized. *Fix:* add `accent-disabled` / `button-loading-bg` tokens to the frontmatter, or specify disabled/loading states as "accent at 40% opacity" with a documented value.

- **low** Browser-chrome lock icon uses `positive` (#3ECF8E) decoratively (all 7 mocks, e.g., key-signin.html line 51). DESIGN.md Don'ts: "Never use them [semantic colors] decoratively" (line 396). Mock furniture only, not product UI, but technically violates the spine's own rule. *Fix:* use `text-2` or a neutral green outside the semantic palette for mock chrome, or accept that browser-chrome furniture is outside the product design system's scope and note it.

- **low** Spinner treatments inconsistent between mocks (key-onboarding.html lines 203–205, key-new-conversation.html lines 225–226). Onboarding spinner: white track at 30% opacity, white head. New-conversation spinner: `border` (#2B2B38) track, `accent` (#7B6EE8) head. Different visual language for the same loading concept. *Fix:* standardize on one spinner treatment (accent head on border track is more on-brand) and document it in DESIGN.md.

## 2. Margins & Spacing — adequate

Checked: 4px grid compliance across all spine component specs and all 7 mockups, spacing-scale tokenization, padding consistency across related components, cross-surface padding consistency, undocumented max-widths.

### Findings

- **high** The 4px grid is violated throughout the mockups and in the spine's own component specs (DESIGN.md line 266: "All internal spacing is a multiple of 4"). Off-grid values found: 7px (nav-item vertical padding — all 5 app-shell mocks), 10px (credential-error-banner padding in DESIGN.md frontmatter line 186; list-entry padding, key-artifact-browser.html line 202; teaser-item padding, key-settings.html line 232; field-group gap, key-onboarding.html line 111), 11px (GitHub button padding, key-signin.html line 130; connect-button padding, key-onboarding.html line 179), 14px (field-input horizontal padding, key-onboarding.html line 133; teaser-item horizontal padding, key-settings.html line 232), 6px (frontmatter-badge padding, key-artifact-browser.html line 271; nav-item active margin, multiple mocks). The 10px value in the credential-error-banner frontmatter spec is particularly notable — it violates the grid rule inside the spine itself, not just in a mock. *Fix:* snap all off-grid values to the nearest 4px multiple (7→8, 10→8 or 12, 11→12, 14→12 or 16, 6→4 or 8). If 2px half-steps are needed for tight controls (tool-pill, status-badge), document the exception in DESIGN.md §Layout & Spacing.

- **medium** `spacing` frontmatter has only `base: 4px` — no scale tokens (DESIGN.md lines 61–63). The Tailwind scale (4, 8, 12, 16, 20, 24, 32, 40, 48, 64) is referenced in prose (line 263) but not in the frontmatter. Downstream code cannot reference `{spacing.4}` or `{spacing.8}` — it must use raw values or Tailwind classes, undermining the single-source-of-truth property and contributing to the off-grid proliferation (finding above). The prior rubric review noted this as "optional, not urgent" (decision log line 74) but the off-grid findings here show the cost is real. *Fix:* add scale tokens (`'1': 4px`, `'2': 8px`, …) to the `spacing` frontmatter.

- **medium** Chat input area bottom padding inconsistent between the two chat surfaces (key-new-conversation.html line 245: `padding: 0 32px 24px`; key-conversation.html line 322: `padding: 0 32px 20px`). 24px vs 20px for the same component on two surfaces. *Fix:* standardize on one value (24px is on the Tailwind scale; 20px is also valid). Document in the `chat-input` component spec.

- **medium** Undocumented max-widths across mocks with no spine tokens or system. Auth card 400px (key-signin.html line 65), onboarding card 480px (key-onboarding.html line 65), chat empty state 480px (key-new-conversation.html line 175), artifact card 720px (key-project-map.html line 216), md-content 720px (key-artifact-browser.html line 301), message row 824px (key-conversation.html line 222, comment says "760 + 64 padding"). The 760px chat max-width is documented (DESIGN.md line 263); the others are not. 400, 480, 720, 760, 824 have no documented relationship. *Fix:* either document a content-width system in DESIGN.md §Layout & Spacing (e.g., narrow 400, standard 480, reading 720, chat 760), or accept the ad-hoc values and note them as per-surface decisions.

- **low** Tool-pill padding (2px 8px) vs semantic-pill padding (4px 10px) — visually related components with different sizing (DESIGN.md frontmatter lines 97, 106). Semantic pill is described as "elevated variant of Tool Pill" (DESIGN.md line 345). When both appear in the same message stream, they render at different heights. 2px is also off the 4px grid. *Fix:* align semantic-pill padding to 2px 8px (matching tool-pill) or 4px 8px; if the larger size is intentional for the "elevated" variant, document the relationship.

- **low** Status-badge padding (2px 8px) differs from semantic-pill padding (4px 10px) — both are pill-shaped badges (DESIGN.md frontmatter lines 141, 106). Inconsistent internal sizing for visually similar components. *Fix:* standardize pill-badge padding across tool-pill, semantic-pill, and status-badge variants.

## 3. Consistency — adequate

Checked: cross-mock component consistency (side nav, chat input, send button, page header), token usage consistency between spine and mocks, component naming, class naming, wordmark/brand presentation, scrollbar behavior, shadow discipline, viewport minimum.

### Findings

- **critical** `access-notice` component has prose but no frontmatter spec (DESIGN.md lines 380–381). The `components:` frontmatter (lines 65–189) defines 16 components; the Access Notice is described in the §Components prose but has no `access-notice` key. Its prose references `{colors.warning}` / `{colors.warning-bg}` which don't exist (see Colors §critical). A downstream builder has no machine-readable spec for this component — only prose pointing to undefined tokens. *Fix:* add an `access-notice:` entry to the frontmatter with resolved token references (background, border, padding, font-size, border-radius), or fold the Access Notice into the `credential-error-banner` spec as a variant if the two are similar enough.

- **high** Nav-item `display` property inconsistent across mocks (key-conversation.html line 114: `display: flex`; key-project-map.html line 103, key-new-conversation.html line 103, key-artifact-browser.html line 103, key-settings.html line 103: `display: block`). The conversation mock uses flex (which vertically centers content with `align-items`), while all other app-shell mocks use block. This changes the vertical alignment and hit-area of nav items. *Fix:* standardize on one display model across all app-shell mocks.

- **high** Conversation mock textarea uses `text-3` (#56556A) for entered text instead of `text-1` (#EDECF5) (key-conversation.html line 344: `.chat-textarea { color: #56556A; }`). The new-conversation mock correctly uses `#EDECF5` (text-1) for the same textarea (key-new-conversation.html line 272). Entered text in the conversation mock renders at 2.29:1 contrast on surface-raised — nearly invisible. If a builder mirrors the mock, typed text would be unreadable. *Fix:* set the conversation mock's textarea color to `#EDECF5` (text-1); reserve `text-3` for the `::placeholder` pseudo-element only.

- **high** Send-button class naming inconsistent (key-new-conversation.html line 294: `.send-button`; key-conversation.html line 360: `.send-btn`). Different class names for the same component across mocks. *Fix:* standardize on one class name; this is a mock-level issue but signals naming drift to a builder reading the mock source.

- **medium** Nav-item active-state margins inconsistent across mocks (key-project-map.html line 117: `margin: 1px 6px`; key-conversation.html line 131: `.nav-item.active` `margin: 0 6px`, line 138: `.nav-item.active-conv` `margin: 1px 6px`; key-settings.html line 138: `margin: 0 6px`). The 1px vs 0px top/bottom margin creates subtle vertical position shifts for active items. The conversation mock has two active classes with different margins. *Fix:* standardize active-state margins across all mocks; use one active class, not two.

- **medium** Slash picker uses `box-shadow`, contradicting the no-shadow rule (key-new-conversation.html line 332: `box-shadow: 0 8px 24px rgba(0,0,0,0.4)`). DESIGN.md Don'ts: "Don't use box shadows to communicate elevation on dark surfaces" (line 399). The slash picker is a floating dropdown that uses shadow for visual separation. The spine doesn't document an exception for floating elements. *Fix:* either document a floating-element exception in DESIGN.md §Elevation & Depth (shadows allowed for overlays/popovers that float above content, not for surface elevation), or replace the shadow with a stronger border or surface contrast.

- **medium** Scrollbars hidden in multiple mocks, undocumented in spine (`::-webkit-scrollbar { display: none; }` in key-project-map.html line 204, key-conversation.html line 217, key-artifact-browser.html lines 199 and 261). The spine doesn't mention scrollbar styling. Hidden scrollbars harm usability (users can't see scroll position) and accessibility (screen-reader users lose scroll context). *Fix:* either document the hidden-scrollbar decision in EXPERIENCE.md (with a `prefers-reduced-motion` or custom-scrollbar fallback), or show scrollbars styled with the palette.

- **medium** Wordmark presentation inconsistent between pre-app-shell and app-shell (key-signin.html lines 231–233: logo box "be" in accent-filled 12px-radius square + "bmad-easy" name + tagline; app-shell mocks e.g. key-project-map.html line 282: text "bmad·easy" with middle dot in accent). Different brand presentations with no documented relationship. *Fix:* document both wordmark treatments in DESIGN.md §Brand & Style (full lockup for pre-app-shell, compact text mark for app-shell) or standardize.

- **low** Dead CSS: `.annotation-band` defined but unused in 4 mocks (key-project-map.html, key-new-conversation.html, key-conversation.html, key-artifact-browser.html all define `.annotation-band` CSS but don't render it in the HTML body). Dead CSS that could confuse a builder reading the mock source. *Fix:* remove unused CSS blocks.

- **low** `min-width: 1280px` on body doesn't match spine's 1024px minimum (all 7 mocks set `min-width: 1280px`; EXPERIENCE.md line 16: "Minimum supported viewport: 1024px wide"). Mocks don't demonstrate the 1024px minimum or the tablet (768–1023px) collapsed-nav behavior. A builder wouldn't know what the UI looks like at 1024px. *Fix:* either render mocks at 1024px (the stated minimum) or add a note that mocks are shown at 1280px for readability and the 1024px layout is spine-only.

## Mechanical notes

- **Broken cross-reference (critical):** `{colors.warning}` and `{colors.warning-bg}` referenced in DESIGN.md line 381 and EXPERIENCE.md line 262 — neither exists in the frontmatter. The prior rubric review (§2, "strong") listed the color tokens it checked but did not include `warning` / `warning-bg`, meaning the reference was introduced after or during the rubric pass and was not re-verified.
- **Missing component frontmatter (critical):** `access-notice` described in DESIGN.md §Components prose (line 380) but absent from the `components:` frontmatter (16 entries; should be 17).
- **Contrast targets unstated:** DESIGN.md §Colors does not state WCAG contrast targets for any text/surface combination. The rubric requires this for load-bearing combinations. `text-3` fails AA everywhere; `accent` as text on `surface` is borderline (4.50:1, exactly at threshold).
- **Spacing scale not tokenized:** `spacing` frontmatter has `base: 4px` only; the Tailwind scale is prose-only. 6 distinct off-grid values (6, 7, 10, 11, 14, 28) appear across mocks and spine specs.
- **Class naming drift:** `.send-button` vs `.send-btn` across two mocks for the same component.
