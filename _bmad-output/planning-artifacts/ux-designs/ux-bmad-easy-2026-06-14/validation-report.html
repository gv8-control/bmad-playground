bmad-easy — UX Design Validation Report

_bmad-output/planning-artifacts/ux-designs/ux-bmad-easy-2026-06-14/

The spine pair is substantially complete and architecturally sound. DESIGN.md is well-disciplined for a shadcn brand-delta document, and EXPERIENCE.md covers the behavioral contract with real depth and precision across states, interaction primitives, and accessibility. The main risks are (1) EXPERIENCE.md's composition-reference links resolve to paths that do not exist in the workspace, making them unverifiable and potentially orphaned, (2) several components appear in both spines under names that are nearly — but not exactly — identical, and (3) the accessibility floor makes a contrast claim against {colors.primary} and {colors.accent} that is stated but not verified in DESIGN.md, which is the authoritative source for contrast data.

These are addressable before handoff to architecture and story-dev without a full rework. The broken visual reference coverage is the highest-priority fix: downstream consumers will attempt to find the four composition files and fail.

1. Flow coverage

adequate

The PRD defines three Key User Journeys: UJ-1 (repository connection), UJ-2 (Skill Session and Commit Pill), UJ-3 (reading a teammate's artifact). EXPERIENCE.md defines four Key Flows. Flows 1–3 map verbatim to UJ-1 through UJ-3. Flow 4 (session expiry recovery) is an extension of UJ-2's edge case ("session idle for 30 minutes") promoted to its own flow — a reasonable judgment call. Each flow has a named protagonist (Sarah throughout), numbered steps, a clearly marked Climax, and at least one failure path where applicable.

medium

Save Checkpoint interaction missing from all flows

EXPERIENCE.md — Key Flows, Flow 2
Flow 2 omits the Working Tree State Indicator / Save Checkpoint interaction, which is the most complex new behavior introduced in FR-18/FR-19/FR-20 and not present in any other flow. No user path shows Sarah noticing "● Unsaved changes", triggering a checkpoint, and seeing the platform-checkpoint Commit Pill.
Fix: Add a Flow 5 (or extend Flow 2 with a branch) showing the dirty-tree → Save Checkpoint → checkpoint Commit Pill sequence.
low

Flow 4 missing concurrent-session-limit failure path

EXPERIENCE.md — Key Flows, Flow 4
Flow 4 climax describes Sarah clicking a link to open a new /bmad-brainstorming session. State Patterns specifies what happens if a concurrent session limit is reached, but Flow 4 does not cross-reference this failure case.
Fix: Add a one-line failure note: "If session limit reached: inline message below replaces the start link; user must close an existing session first."
low

Flow 3 has no failure path

EXPERIENCE.md — Key Flows, Flow 3
UJ-3 is fully covered by Flow 3, but Flow 3 has no failure path. No failure path (e.g., artifact deleted from repo between page load and click) is described anywhere in either spine.
Fix: Add a failure note to Flow 3: "If artifact not found or read fails, show a muted error in place of the rendered content with a link back to Project Map."

2. Token completeness

strong

All tokens defined in DESIGN.md frontmatter were traced: colors.primary, colors.primary-foreground, colors.accent, colors.accent-foreground, colors.primary-dark, colors.primary-foreground-dark, colors.accent-dark, colors.accent-foreground-dark; typography.mono.fontFamily; rounded.sm, rounded.md, rounded.lg, rounded.pill; spacing (semantic note, no override). All {path.to.token} references in component objects and prose resolve to their frontmatter definitions. Dark-mode pairs are present for both brand colors. No hex is missing.

medium

Contrast claim in EXPERIENCE.md not verified in DESIGN.md

EXPERIENCE.md — Accessibility Floor; DESIGN.md — Colors
EXPERIENCE.md states "{colors.primary} and {colors.accent} verified to maintain WCAG AA contrast ratios against their respective background tokens." DESIGN.md does not include a contrast annotation. {colors.accent} (#EA5C1E) on white is approximately 3.1:1 — passes AA for large text only, fails for body-size text.
Fix: Add a contrast table to DESIGN.md Colors section listing load-bearing combinations: primary on background (light/dark), accent on muted, accent-foreground on accent. Flag the accent/white combination as large-text-only if that holds at final hex.
low

working-tree-indicator uses Tailwind class name instead of token value

DESIGN.md — frontmatter, working-tree-indicator component
The component uses font: 'Geist Sans' (re-states the inherited default) and font-size: 'text-xs' (a Tailwind class name, not a token value). Minor type inconsistency with the rest of the frontmatter.
Fix: Remove the font key (it's the default) and change font-size: 'text-xs' to a semantic note or a px dimension for consistency.
low

session-status-expired missing font token reference

DESIGN.md — frontmatter, session-status-expired component
The frontmatter entry specifies only color. The prose description adds "Geist Mono" for the label font, but the frontmatter component is missing that font token reference.
Fix: Add label-font: '{typography.mono.fontFamily}' to the session-status-expired frontmatter entry to match the prose description.

3. Component coverage

adequate

11 components have both visual spec (DESIGN.md) and behavioral spec (EXPERIENCE.md): Button (primary variant), Commit Pill (both variants), Skill Launch Field, Session Status (active/expired), Agent Thinking Indicator, Agent Tool Indicator, Chat Bubble — User, Chat Bubble — Agent, Artifact Row, Credential Warning Banner, Working Tree State Indicator. Several utility components (Manual Refresh, Stop Button, Scroll-to-Bottom) have behavioral specs but no visual spec; the expected shadcn inheritance is not made explicit.

medium

Save Checkpoint confirmation UI has no visual spec

EXPERIENCE.md — Component Patterns; DESIGN.md — Components
The inline Save Checkpoint confirmation UI is described behaviorally in EXPERIENCE.md but has no visual spec. Story-dev cannot know whether this is a Dialog, an inline form, a popover, or a simple confirm/cancel pair within the chat input area.
Fix: Add a visual spec to DESIGN.md Components for save-checkpoint-confirmation (e.g., inline below Working Tree State Indicator, muted background, rounded.md, confirm + cancel buttons using shadcn defaults) or explicitly state it inherits shadcn AlertDialog.
medium

session-status-provisioning has no frontmatter entry

DESIGN.md — frontmatter, Components prose
session-status-provisioning is described in DESIGN.md prose ("shadcn spinner + 'Starting session…' in Geist Sans muted") but has no frontmatter entry. The other two Session Status states have frontmatter entries. This leaves the provisioning state underspecified for token reference.
Fix: Add session-status-provisioning: { label-font: 'Geist Sans', color: 'hsl(var(--muted-foreground))' } to the frontmatter. The spinner is a shadcn default and needs no override.
low

DESIGN.md — Components
All three have behavioral specs but no visual specs. shadcn inheritance is the likely intent but is not stated.

low

DESIGN.md has separate frontmatter keys commit-pill and commit-pill-checkpoint, but EXPERIENCE.md describes both in a single "Commit Pill" row. Story-dev creating implementation tickets will see a name mismatch.

medium

Artifact Browser missing load-failure state

EXPERIENCE.md — State Patterns, Artifact Browser
No state is defined for when an artifact cannot be loaded (git read failure, artifact path deleted, network error after page load). The loading state is covered but the failure state is absent. Story-dev will implement something without a spec for what.
Fix: Add a row: "Artifact load failure | Artifact Browser | Muted inline error message: 'This artifact could not be loaded.' Link: '← Project Map'. No retry affordance."

Onboarding missing loading state between Connect click and validation result

EXPERIENCE.md — State Patterns, Onboarding
Flow 1 describes a brief shadcn Skeleton state during PAT + repository validation, but this state is in the Key Flows narrative only — not in State Patterns, which lists only error and success outcomes for Onboarding step 2.
Fix: Add a row: "Validating PAT + repository | Onboarding | Inline shadcn Skeleton / spinner on the Connect button. Button disabled while validation is in progress (prevents duplicate submissions)."

EXPERIENCE.md — State Patterns, Skill Session

medium

No visual references exist for any of the six IA surfaces

mockups/, wireframes/, imports/ — all empty or absent
For a project where the non-dev user mental model is central to the product premise, absence of any visual reference is a risk for story-dev — component tables and prose may be interpreted differently by engineers than the author intended. The Skill Session layout in particular (Thinking Indicator, Tool Indicator, Working Tree State Indicator, Stop Button, chat input with multiple affordances) is complex enough that a layout reference would reduce interpretation divergence.
Fix: A low-fidelity annotated sketch or ASCII layout of the Project Map and Skill Session page surfaces would anchor the composition for story-dev. Not spec-required but strongly advisable.

6. Bloat & overspecification

strong

DESIGN.md is lean and disciplined. It states the brand delta, provides rationale for each decision, and explicitly defers to shadcn defaults everywhere it hasn't earned the override. The Do's and Don'ts table encodes real constraints, not restatements of rules already in the prose. There is no restatement of PRD personas or source material reproduction. EXPERIENCE.md is similarly well-structured; Voice and Tone examples are real microcopy strings, and Component Patterns provide real behavioral rules, not one-word descriptions.

low

"Only if the working tree is dirty" stated twice in State Patterns

EXPERIENCE.md — State Patterns, "Session approaching idle timeout"
The dirty condition is established at the start of the entry and repeated in the sentence describing the pulse. The repetition adds no information.
Fix: Remove the second instance of the qualifier from: "When the BMAD Agent becomes idle (finishes a response) and the working tree is dirty, the status indicator pulses once."

7. Inheritance discipline

adequate

Both spines declare sources: [_bmad-output/planning-artifacts/prds/prd-bmad-easy-2026-06-14/prd.md] — correct path. Glossary terms from the PRD (12 terms) are used verbatim in both spines, with no synonyms introduced. Key Flow names ("Flow 1–4") do not match source UJ codes ("UJ-1–3"), reducing story-dev traceability. Token reference style is inconsistent across EXPERIENCE.md.

medium

Key Flow names don't match PRD UJ identifiers

EXPERIENCE.md — Key Flows; PRD — UJ-1, UJ-2, UJ-3
EXPERIENCE.md uses "Flow 1," "Flow 2," "Flow 3," "Flow 4." The PRD uses "UJ-1," "UJ-2," "UJ-3." Story-dev tracing a ticket to source will need to manually align these, increasing the chance of errors.
Fix: Either name flows "Flow 1 — UJ-1: Repository connection" (append source UJ code) or add a mapping note at the top of Key Flows: "Flows 1–3 correspond to PRD UJ-1, UJ-2, UJ-3 respectively. Flow 4 extends UJ-2's session-expiry edge case."
medium

Token reference style inconsistent within EXPERIENCE.md

EXPERIENCE.md — Accessibility Floor vs. all other sections
The Accessibility Floor uses {colors.primary} and {colors.accent} token syntax. All other EXPERIENCE.md color references use plain English ("ember," "muted foreground," "indigo"). This inconsistency means a token-aware renderer would need to process the file differently, and signals incomplete adoption of the cross-spine token system.
Fix: Pick one approach and apply consistently: either convert all EXPERIENCE.md color mentions to token references, or convert the Accessibility Floor reference to plain English ("primary indigo and ember accent").
low

Save Checkpoint / Working Tree State Indicator / "Save control" — three names for related concepts

EXPERIENCE.md — Foundation, State Patterns, Component Patterns; PRD — FR-19
The multi-name situation is logical (the indicator is the affordance; Save Checkpoint is the operation; "Save control" is the informal PRD handle), but story-dev may create tickets with mismatched names.
Fix: Establish a canonical name at the top of the Component Patterns Save Checkpoint entry: "Save Checkpoint (triggered via Working Tree State Indicator when dirty; also referred to as 'the Save control' in PRD FR-19)."

8. Shape fit

strong

DESIGN.md sections are present and in canonical order: Brand & Style → Colors → Typography → Layout & Spacing → Elevation & Depth → Shapes → Components → Do's and Don'ts. All eight sections present. EXPERIENCE.md required defaults are all present: Foundation, Information Architecture, Voice and Tone, Component Patterns, State Patterns, Interaction Primitives, Accessibility Floor, Key Flows. Responsive & Platform is present (triggered by multi-surface mention and breakpoints). Inspiration & Anti-patterns is absent — see finding.

low

Inspiration & Anti-patterns section absent despite design rationale worth documenting

EXPERIENCE.md — missing section
The product brief calls out Claude Code / similar tools as the source of the / slash-command convention, and the product makes deliberate choices (no vim-style keyboard navigation, Geist Mono signal, non-terminal design) that would benefit from attribution. Without a decision log in the UX workspace, these design decisions are not recorded and are at risk of scope creep later.
Fix: Add an Inspiration & Anti-patterns section with 4–6 entries matching the Drift example format — what was borrowed and from where, and what was explicitly rejected (e.g., "no vim-style keyboard navigation — primary audience is PMs, not developers").

Mechanical notes

  • Frontmatter completeness: DESIGN.md has name, description, status, created, updated, sources, and all token sections. EXPERIENCE.md is missing description — minor, not required by spec, but present in DESIGN.md and the example spines.
  • No Mermaid diagrams present in either spine. Not required; no syntax to check.
  • Cross-spine token references: EXPERIENCE.md uses {colors.primary} and {colors.accent} in the Accessibility Floor (line 123) — these resolve correctly to DESIGN.md frontmatter. Only two token references appear in EXPERIENCE.md total; all others are plain English.
  • Name consistency: "Skill Launch Field" consistent across both spines (PRD FR-9 uses "skill entry field" once, but spines are consistent). "Commit Pill" / "commit-pill" consistent. "Checkpoint" vs "platform checkpoint" vs "Save Checkpoint" — three names in use; see §7 finding.
  • No .decision-log.md in UX workspace: Design decisions (rejection of vim-style keyboard shortcuts, promotion of Flow 4, omission of Inspiration & Anti-patterns) are not logged. The PRD workspace has a decision log but the UX workspace does not.
